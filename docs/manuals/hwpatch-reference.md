# FITOM_X HwPatch リファレンス

「HwPatch」は、FM音源チップ・PSGチップのレジスタへ直接書き込まれる、音色そのものの合成パラメータです(フィードバック、アルゴリズム、各オペレータのエンベロープ等)。チップの種類によって、実際に効くフィールド・値の意味が異なります。本書は、チップを選んだときに実際に使われるフィールドとその意味をまとめたリファレンスです。

演奏特性(ビブラート・ベロシティ感度等)は[SwPatch リファレンス](./swpatch-reference.md)、ネイティブパッチ全体の構成は[ネイティブパッチ リファレンス](./native-patch-reference.md)を参照してください。MIDI経由でこれらのパラメータを演奏中に上書きする方法は[MIDIメッセージリファレンスマニュアルの「8.1 パッチパラメータオーバーライド」](./midi-message-reference.md#81-パッチパラメータオーバーライドプライベートsysex非標準)を参照してください。

---

## 1. 共通フィールド構成

1つのHwPatchは、以下の3つのグループのフィールドを持ちます。

```
(チャンネル単位、パッチ1つにつき1組): FB, ALG, AMS, PMS, NFQ, FB2
ops[] (オペレータ単位、最大4個):      AR, DR, SL, SR, RR, TL, KSR, KSL, MUL,
                                       DT1, DT2, PDT, AM, VIB, EGT, WS, REV, EGS, DT3
ext (チップ固有拡張、パッチ1つにつき1組): FIX, ALG_EXT, HWEP,
                                           target_voice_patch_type, rhythm_ch
sw_bank / sw_prog: このHwPatch自身が指定するパフォーマンスパッチの参照先
```

`ops[]`の要素数は、チップのオペレータ数に応じて1〜4個で可変です(1=SSG/DCSG/SAA/SCC等のPSG系、2=OPL/OPL2/OPL3の2opモード/OPLL系、4=OPN/OPN2/OPM/OPZ/OPZ2/OPL3の4opモード)。指定した要素数分だけが読み込まれ、残りはデフォルト値のまま扱われます。

`ext`・`FB2`・`PDT`等の多くのフィールドは「特定のチップのみが参照し、他のチップでは無視される(0のままで問題ない)」設計になっています。以下、チップごとに実際に意味を持つフィールドのみを記載します。

### エンベロープ系フィールドの値域について(重要)

`AR`/`DR`/`SL`/`SR`/`RR`/`TL`は、JSONスキーマ(`hwbank.schema.json`)上は
全チップ共通で以下の範囲を受け付けます。

| フィールド | スキーマ上の範囲 |
|---|---|
| `AR`(Attack Rate) | 0–31 |
| `DR`(Decay Rate) | 0–31 |
| `SL`(Sustain Level) | 0–15 |
| `SR`(Sustain Rate、FITOM独自拡張。詳細は各チップ節参照) | 0–31 |
| `RR`(Release Rate) | 0–15 |
| `TL`(Total Level) | 0–127 |

しかし、**チップの実機レジスタ幅がこれより狭い場合、下位ビットは実際には
無視されます**(値を指定しても、複数の値が同じ結果になります)。以下の
各チップ節では、それぞれの実効ビット幅と、その結果何刻みで値が区別できる
かを明記します。値を細かく調整しても音が変わらない場合は、まずこの実効
解像度を確認してください。

## 2. OPN(YM2203)

| フィールド | 範囲 | 内容 |
|---|---|---|
| `FB` | 0–7 | フィードバック |
| `ALG` | 0–7 | アルゴリズム |
| `ops[0-3].KSR/MUL/DT1` | 各項目参照 | 通常のFM 4オペレータパラメータ |
| `ops[i].EGT` | 0–15 | SSG-EG(OPN/OPNA系のみ) |

**エンベロープ実効解像度**: `AR`/`DR`/`SL`/`SR`/`RR`/`TL`はスキーマ上の
範囲(0–31/0–15/0–127)を実機レジスタがそのままフルビット幅で持つため、
**指定した値がそのまま1刻みで反映されます**(丸めなし)。`SR`はOPN実機の
"D2R"(第2ディケイレート)そのもので、OPL系のような「パーカッシブ/
サステイン切替」という特別な意味は持たず、通常のADSR(AR→DR→SL→SR→RR)の
一部として機能します。

### FXモード(ch2専用、実機の「3rd channel special mode」)

OPNのチャンネル2(3番目のチャンネル)は、4つのオペレータそれぞれに個別の音程を持たせる特殊モードを持ちます。

| フィールド | 内容 |
|---|---|
| `ext.FIX` | モード選択。0=通常(FXモード無効)/1=疑似デチューン/2=非整数倍率/3=固定周波数 |
| `ops[i].PDT` | モード1・2:100/64セント単位の符号付きオフセット。モード3:0.1Hz単位の絶対周波数 |

ch2以外、および`FIX=0`のパッチには影響しません。

## 3. OPM(YM2151)/OPP(YM2164)

| フィールド | 範囲 | 内容 |
|---|---|---|
| `FB` / `ALG` / `AMS` / `PMS` / `NFQ` | 各項目参照 | 通常のFMパラメータ |
| `ops[0-3].KSR/MUL/DT1/DT2/AM` | 各項目参照 | 通常のFM 4オペレータパラメータ |

### エンベロープ実効解像度

`AR`/`DR`/`SR`はスキーマ通りフル5bit(0–31、1刻み)、`SL`/`RR`はフル4bit
(0–15、1刻み)、`TL`はフル7bit(0–127、1刻み)が、実機YM2151の対応する
レジスタ幅(順にAR/D1R/D2R=5bit、D1L/RR=4bit、TL=7bit)そのままに反映され
ます(2026年7月、レジスタ書き込み時に余分なビットシフトが入り実効解像度
が大幅に劣化していた実装バグを修正。修正前はAR/DR/SRが実質8段階、SL/RR
が実質2段階にまで縮退していました)。

### ノイズモード(ch7専用)

| フィールド | 内容 |
|---|---|
| `ext.ALG_EXT`(bit0) | 1=ノイズ有効。ノイズを要求すると自動的にch7へ割り当てられます。 |

| フィールド | 範囲 | 内容 |
|---|---|---|

| `ops[i].EGS` | 0–127 | EGバイアス(オペレータ単位) |
| `ops[i].WS` | 0–7 | 波形選択(OPZ独自波形) |

## 5. OPL(YM3526)/OPL2(YM3812)

| フィールド | 範囲 | 内容 |
|---|---|---|
| `FB` | 0–7 | フィードバック |
| `ALG`(bit0のみ) | 0–1 | 0=FM/1=AM(2オペレータのみ) |
| `ops[0-1].SL/KSR/KSL/MUL/AM/VIB/EGT` | 各項目参照 | 通常の2オペレータパラメータ |
| `ops[i].WS` | 0–3(OPL2以降) | 波形選択 |

### エンベロープ実効解像度

`AR`/`DR`は実機が4bitしか持たないため、スキーマ上の範囲(0–31)のうち
**上位4bitのみが有効**です。2刻み(偶数/奇数で同じ値になる)で変化し、
実際に区別されるのは16段階です(`AR=14`と`AR=15`は同じ結果になります)。

`TL`も実機が6bitしか持たないため、スキーマ上の範囲(0–127)のうち上位
6bitのみが有効です。同様に2刻みで、実際に区別されるのは64段階です。

`SL`/`RR`はスキーマ通りフル4bit(0–15、1刻み)がそのまま反映されます。

### `SR`(サステインレート)と`RR`(リリースレート)の関係

OPL系の実機は「サステインあり」と「パーカッシブ(サステインなし)」という2つの動作モードを切り替える仕組みを持ちます。FITOM_Xでは、この違いを`SR`フィールドの有無で統一的に表現します。

- `SR`に0より大きい値を指定すると、パーカッシブモード(キーオン中もその値の速さで減衰し続ける)になります。この場合、実機RRレジスタには`RR`の代わりに`SR`の値が書き込まれます(`SR`もAR/DRと同じく上位4bitのみ有効、2刻み)。
- `SR=0`の場合、通常のサステインモード(`RR`の値でキーオフ後に減衰)になります。
- この切り替えはキーオン/キーオフのたびに動的に行われます(音色をロードした瞬間ではなく、実際に鍵盤を押す/離すタイミングで反映されます)。

## 6. OPL3(YMF262)4オペレータモード

| フィールド | 範囲 | 内容 |
|---|---|---|
| `ALG`(bit0-2) | 0–7 | bit0=前半ペア接続、bit1=後半ペア接続、bit2=ConnectionSEL(4オペレータ結合の有効化。前半・後半ペアを1つの音として結合するかどうか) |
| `FB` | 0–7 | 前半ペア(オペレータ1・3)のフィードバック |
| `FB2` | 0–7 | 後半ペア(オペレータ2・4)のフィードバック(実機は前半・後半で別々のフィードバックを持てます) |
| `ops[0-3].SL/KSR/KSL/MUL/WS/AM/VIB/EGT` | 各項目参照 | 通常の4オペレータパラメータ |

`AR`/`DR`/`TL`の実効解像度(上位4bit/6bitのみ有効、2刻み)、および`SR`/`RR`の関係は「5. OPL/OPL2」と同じです。

### ConnectionSEL(`ALG`のbit2)とキーオンの関係

実機YMF262は、`ALG`のbit2(ConnectionSEL)が1(4オペレータ結合)の間、
**後半ペア自身のレジスタ(Key-On/Block/F-Numberを含む)を無視**し、前半
ペアの値だけで4オペレータ全体を駆動します(ゆえに結合時は前半ペアの
キーオンのみが実際に意味を持ちます)。逆にbit2が0(前半・後半が独立した
2オペレータ×2)の場合は、両ペアのキーオンがそれぞれ独立に意味を持つため、
後半ペアも鳴らしたい場合は両方にキーオンを送る必要があります。

### 疑似デチューン

| フィールド | 内容 |
|---|---|
| `ops[0].PDT` | 前半ペア用、100/64セント単位の符号付きオフセット |
| `ops[2].PDT` | 後半ペア用、同上 |

`ops[i].DT1`/`DT2`はOPL系チップには対応する仕組みが無いため、常に無視されます(0のままで構いません)。

## 7. OPLL系(YM2413/YM2420/YMF281/YM2423/FS1001)

| フィールド | 範囲 | 内容 |
|---|---|---|
| `FB` / `ALG` | 各項目参照 | 通常パラメータ(2オペレータ) |
| `ops[0-1].SL/KSR/AM/VIB/EGT` | 各項目参照 | 通常の2オペレータパラメータ |
| `ops[0].TL` | 0–127(実効6bit、2刻み) | モジュレータレベル。**オペレータ0(モジュレータ)側のTLです。無視されません**(下記参照)。 |
| `ops[1].TL` | 0–127(実効4bit、dB変換) | キャリア(オペレータ1)の音量。実機が4bitの音量レジスタしか持たないため、`linear2dB()`で48dB/1.5dBステップの16段階に変換されます(単純な下位ビット切り捨てではなく対数変換のため、大きい値側ほど1段階あたりの音量差が大きくなります)。 |

`AR`/`DR`の実効解像度(上位4bit、2刻み)、および`SR`/`RR`の関係は「5. OPL/OPL2」と同じです。

### プリセット/ユーザー音色

| フィールド | 内容 |
|---|---|
| `ext.ALG_EXT`(bit0) | 1=プリセット音色(チップ内蔵ROM、パラメータ変更不可)、0=ユーザー音色 |

### 内蔵リズム(COPLLRhythm、5パート)

内蔵リズム音源としての使い方はMIDIの直接デバイス選択値`0x70`(内蔵リズム音源専用バンク)経由で行います。詳細は[MIDIメッセージリファレンスマニュアル](./midi-message-reference.md)の該当節を参照してください。

## 8. PSG系共通(SSG/EPSG/DCSG/SAA/SCC)

全PSG系チップが共通で使うフィールドです。実機の細かい挙動はチップごとに異なります。

| フィールド | 範囲 | 内容 |
|---|---|---|
| `ops[0].AR/DR/SL/SR/RR` | スキーマ通りフル(0–31/0–15、1刻み) | ソフトウェアエンベロープ(チップにハードウェアエンベロープが無い、または未使用の場合に使われます) |
| `ops[0].TL` | 0–127 | 基準音量 |
| `ops[0].EGT`(bit3=0x08) | 0/8 | ハードウェアエンベロープ使用フラグ(1=ハードウェアエンベロープ、0=ソフトウェアエンベロープ)。DCSG/SCCは実機にハードウェアエンベロープが無いため、このビットは無視されます。 |

ソフトウェアエンベロープは実機レジスタの制約を受けないため、`AR`/`DR`/`SL`/`SR`/`RR`はFM系チップのような実効解像度の低下(丸め・上位ビットのみ有効等)がなく、スキーマ上の値がそのまま1刻みで反映されます。ただしAY8930(EPSG)のハードウェアエンベロープモードのみ例外があります(「12. AY8930(EPSG)」参照)。

PSG系は全チップが共通のバンク空間(直接デバイス選択値`0x40`固定)を共有する設計になっています。

## 9. SSG(YM2149/AY-3-8910)

| フィールド | 範囲 | 内容 |
|---|---|---|
| `ALG`(下位2bit) | 0–3 | 0=トーンのみ/1=ノイズのみ/2=両方/3=両方無効(消音) |
| `NFQ` | 0–31 | ノイズ周波数(ノイズを使う場合のみ) |
| `ext.HWEP` | 0–65535 | ハードウェアエンベロープ周期(ハードウェアエンベロープ使用時のみ) |
| `ops[0].EGT`(下位4bit) | 各項目参照 | エンベロープ波形シェイプ(ハードウェアエンベロープ使用時) |

ハードウェアエンベロープ使用時(`EGT`bit3=1)は`ops[0].AR/DR/SL/SR/RR`は完全に無視され、`ext.HWEP`と`EGT`下位4bitのみで減衰カーブが決まります(8. PSG系共通のソフトウェアエンベロープとは排他的な別経路)。

## 12. AY8930(EPSG)

SSGと同じ基本的な意味論に加え、拡張機能を持ちます。

| フィールド | 範囲 | 内容 |
|---|---|---|
| `ALG`(下位2bit) | 0–3 | SSGと同じ(0=トーン/1=ノイズ/2=両方/3=両方無効) |
| `NFQ` + `FB` | 各項目参照 | 拡張ノイズ周波数(8bit) |
| `ops[0].WS` | 0–15 | デューティ比(矩形波のパルス幅、AY-3-8910には無い拡張機能) |
| `ops[0].EGT`/`SL`/`RR`/`DR`/`SR` | 各項目参照 | ハードウェア/ソフトウェアエンベロープ切替 |

**ハードウェアエンベロープ使用時の`DR`/`SR`実効解像度**: ソフトウェア
エンベロープ時はSSGと同じくフル解像度(9. SSG参照)ですが、AY8930拡張の
ハードウェアエンベロープモードでは`DR`/`SR`はレジスタの下位4bitにしか
書き込まれません(`& 0xF`で上位1bitが切り捨てられ、16で折り返します。
例: `DR=3`と`DR=19`は同じレジスタ値になります)。`SL`/`RR`は元々4bit
(0–15)のため欠落はありません。

## 13. SAA1099

実機データシートに基づく実装です。

| フィールド | 範囲 | 内容 |
|---|---|---|
| `ALG`(下位2bit) | 0–2 | 0=トーンのみ/1=ノイズのみ/2=両方 |
| `NFQ`(下位2bit) | 各項目参照 | ノイズパラメータ(チャンネル0-2/3-5の2系統で共有) |
| `ops[0].EGT`(bit3) | 0/1 | 1=ハードウェアエンベロープ使用(チャンネル0-2、または3-5の3ch単位で共有) |
| `ext.HWEP`(下位6bit) | 各項目参照 | ハードウェアエンベロープの波形・分解能・クロックソース・右チャンネル反転をビットで指定 |

ハードウェアエンベロープは3ch単位で共有される実機制約があるため、既に同じエンベロープ設定で発音中のチャンネルがあるグループが優先的に選ばれます。パンポットは自動的に等パワーパンニングで計算されるため、パッチ側での指定は不要です。

## 14. ADPCM-B / ADPCM-A / PCM-D8 / AWM(サンプルベース音源系共通)

**これらのチップはいずれも専用のデータ形式(SampleZonePatch)を使い、HwPatchの`ops[]`は一切使いません。** 「1プログラム=複数の鍵盤域(キーゾーン)への波形/サンプルマッピング」という、FMオペレータ型のパラメータとは本質的に異なる形のデータのためです(`isSampleBasedVoicePatchType()`、`PatchManager::resolveDirect()`のADPCM/AWM共通分岐)。

| フィールド(`zones[]`の各要素) | 内容 |
|---|---|
| `key_min` / `key_max` | このゾーンが適用されるMIDIノート範囲 |
| `vel_min` / `vel_max` | ベロシティレイヤー範囲(省略時0–127=無制限) |
| `wave_index` | チップ側のROM/PCM波形番号(チップ依存の生値)。AWMなら内蔵ROM波形の番号、ADPCM-B/ADPCM-A/PCM-D8ならPCM波形バンク(`*.pcmbank.json`)内のエントリ番号 |
| `root_note` | 録音時の基準ノート(MIDI note番号)。ピッチ可変のPCM系チップがノートからの相対ピッチシフト量を計算するのに使う。省略時69(A4) |

ノートオン時、`zones[]`を先頭から探索し、ノート番号とベロシティの両方が範囲内に収まる最初のゾーンが使われます。専用のバンクファイル形式(`*.samplezonebank.json`)で記述するか、ADPCM-B/ADPCM-A/PCM-D8の場合は下記の自動合成を使えます。

### ADPCM-B/ADPCM-A/PCM-D8: pcmbank.jsonからの自動合成

個々のパッチ(`zones`が1件だけの最小限のSampleZonePatch、`wave_index`=PCM波形バンクのエントリ番号)は、`*.samplezonebank.json`を手書きしなくても自動生成できます。プロファイルの`banks.pcm_banks[]`エントリに`"group": "ADPCMB"` / `"ADPCMA"` / `"PCMD8"`を指定すると、参照先の`*.pcmbank.json`(`entries[]`、adpcm_packer出力由来)の各サンプルから、`prog`=エントリ番号・`name`=サンプル名・`zones[0].wave_index`=同じエントリ番号・`zones[0].root_note`=adpcm_packer出力の`root_note`(省略時69)というnamed patchを`PatchManager::loadPcmBankJson()`が自動的に合成し、通常の`*.samplezonebank.json`経由のパッチと同様にパッチピッカー(CC#0=ADPCM-B/ADPCM-A/PCM-D8)から選択・試聴できるようにします。`group`を省略した場合は波形データの登録のみが行われ、named patchは合成されません(後方互換)。

---

## 15. `sw_bank` / `sw_prog`(パフォーマンスパッチ参照)

すべてのHwPatchに共通のフィールドです。このHwPatch(音色)が「本来意図している」演奏特性([SwPatch リファレンス](./swpatch-reference.md)参照)を指定します。-1(省略時のデフォルト)は「参照なし」を意味します。ネイティブパッチ側(パッチ単位の`sw_bank`/`sw_prog`)で個別に指定されている場合は、そちらが優先されます。指定先が見つからない場合はパフォーマンスパッチが適用されないだけで、HwPatch自体の発音は妨げられません。
